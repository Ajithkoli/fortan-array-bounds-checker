Semantic Validation Analysis

During this final compiler pass, Flang's semantic diagnostics emit compile-time errors or warnings directly. Absolute violations (OOB errors) halt compilation, while unresolved variable subscripts emit warnings to advise additional runtime bounds verification.

📊 Flang Bounds-Checker Plugin vs gfortran -fbounds-check

Both tools work on Fortran source. The key difference: Flang's plugin catches violations at compile time (zero runtime cost), while gfortran inserts runtime guards that fire when the program executes.

Fortran Bounds-Checking Strategy Comparison

Feature Flang Plugin (This Project) gfortran -fbounds-check
When it catches OOB 🕐 Compile time — before the program runs ⚡ Runtime — when the bad access actually executes
Mechanism Hooks into Flang semantic pass via CheckHelper::Enter(ArrayElement&) Inserts branch + trap instructions around every array subscript in the binary
Constant index (e.g. A(11)) ✅ Error — folded and checked at compile time ✅ Caught at runtime (wastes a branch that could have been compile-time)
Variable index (e.g. A(i)) ⚠️ Warning — cannot verify statically ✅ Checked at runtime (with overhead per access)
Constant expression (e.g. A(5+6)) ✅ Folds 5+6 → 11, emits Error at compile time ✅ Caught at runtime after fold
Arbitrary bounds (e.g. B(-5:5)) ✅ Fully supported — lower/upper both checked ✅ Fully supported at runtime
Runtime performance overhead 🟢 Zero — all analysis at compile time 🔴 Significant — every access pays a branch cost
